Building bilingual Arabic & English apps for the Gulf

4 June 2026 · 2 min read · Khatib Designs

For most GCC products, supporting Arabic isn’t a nice-to-have — it’s the difference between an app that feels local and one that feels imported. But true bilingual support is much more than running text through a translator.

RTL is a layout decision, not a text one

Arabic reads right-to-left, which means the entire interface should mirror: navigation, icons, back buttons, progress bars, sliders and alignment all flip. Done properly, an Arabic user feels the app was built for them — not bolted on. Done badly, it feels broken.

Localisation > translation

Good localisation respects more than words:

  • Numbers, dates and currency formatted the way locals expect.
  • Tone and phrasing that read naturally in Arabic, not literal translations.
  • Typography — Arabic type needs its own care for line height, weight and legibility.
  • Imagery and examples that feel regionally relevant.

Build it in from the start

The cheapest time to add Arabic is at the beginning. Retrofitting RTL into an app that was only ever designed for English is slow and error-prone; designing for both from day one is smooth. We build language switching and full RTL support into the foundation — as we did on healthcare and commerce products serving Gulf users in both languages.

Frequently asked questions

Is adding Arabic to an app just translation?

No. True Arabic support means mirroring the whole layout for right-to-left (RTL), localising numbers, dates and tone, and using proper Arabic typography — not just translating text.

Should I add Arabic from the start or later?

From the start. Designing for Arabic and English together is smooth; retrofitting RTL into an English-only app later is slow and costly.
